# 批量任务模板项目属性方案

## 概述

为批量任务模板增加 `projects: string[]` 属性，用于限定模板可被哪些项目使用：
- `projects` 为空数组 → 该模板可被**所有**项目使用（向后兼容）
- `projects` 非空 → 该模板仅可被列表中**指定**的项目使用

在「批量创建任务」流程中，根据当前项目过滤模板列表，仅展示可用模板。

本方案基于 `plans/批量任务模板增加项目属性的方案.md` 的原始思路，结合实际代码结构（行号、签名、i18n 三语、测试约定）做了落地细化。

### 项目名来源

项目名 = `tasksRootPath` 下的**文件夹名**（`ProjectFolderEntry.name`），由 `listProjectFolders` 扫描得到。设置 Tab 可通过 `this.plugin.settings.tasksRootPath` + `listProjectFolders(this.app, ...)` 获取当前所有项目名，用于编辑弹窗的候选列表。

## 拟定变更

### 变更 1：扩展数据结构与纯函数

**文件**：`src/tasks-center/batch-task-template.ts`

1. **`BatchTaskTemplate` 接口**（第 20-25 行）增加字段：
   ```typescript
   export interface BatchTaskTemplate {
       id: string;
       name: string;
       levelTaskTypes: BatchTaskType[];
       listContent: string;
       projects: string[]; // 新增：所属项目列表，空数组=所有项目可用
   }
   ```

2. **`normalizeBatchTemplate`**（第 237-258 行）解析并规范化 `projects`，返回值加入该字段：
   ```typescript
   const projects = Array.isArray(candidate.projects)
       ? candidate.projects
             .map((p) => (typeof p === 'string' ? p.trim() : ''))
             .filter((p) => p.length > 0)
       : [];
   // ...
   return { id, name, levelTaskTypes, listContent, projects };
   ```
   - 旧数据无 `projects` → 自动得 `[]`，向后兼容。

3. **`areBatchTemplateConfigsEqual`**（第 297-322 行）的逐模板比较中加入 `projects` 比较。由于项目选择顺序不影响语义，采用「排序后逐项比较」：
   ```typescript
   function areProjectsEqual(left: string[], right: string[]): boolean {
       if (left.length !== right.length) return false;
       const a = [...left].sort();
       const b = [...right].sort();
       return a.every((v, i) => v === b[i]);
   }
   ```
   在第 312-320 行的返回表达式中追加 `&& areProjectsEqual(template.projects, other.projects)`。

**为什么**：这是数据层的根基；`normalizeBatchTemplate` 保证旧配置兼容，`areBatchTemplateConfigsEqual` 保证「编辑 projects 后能被识别为变更并保存」（否则 `updateBatchTemplateConfig` 会误判无变化而不落盘）。

### 变更 2：编辑弹窗支持选择项目

**文件**：`src/ui/batchTemplateEditModal.ts`

1. **构造函数扩展**（第 47-57 行）新增可选参数 `availableProjects`：
   ```typescript
   constructor(
       app: Modal['app'],
       existing: BatchTaskTemplate | null,
       availableProjects: string[] = [],
   )
   ```
   - 新增私有字段 `private projects: string[] = [];` 与 `private readonly availableProjects: string[] = [];`
   - 构造体内：`this.projects = existing?.projects ?? [];`

2. **`onOpen` 增加「所属项目」设置**（插入到名称 Setting 之后、`levelTypesContainerEl` 之前，即第 81 行与第 83 行之间）：
   - 使用 `TextAreaComponent`，每行一个项目名（与原方案一致，简单直观）。
   - 候选项目作为 placeholder 提示（例如显示前几个项目名）。
   - `onChange` 解析换行 → trim → 过滤空行 → 写入 `this.projects`。
   - 文案键：`settings.batchTemplates.editModal.projects` / `projectsDesc` / `projectsPlaceholder`。

   实现要点：
   ```typescript
   new Setting(this.contentEl)
       .setName(t('settings.batchTemplates.editModal.projects'))
       .setDesc(t('settings.batchTemplates.editModal.projectsDesc'))
       .addTextArea((textArea) => {
           textArea.setValue(this.projects.join('\n'));
           textArea.setPlaceholder(
               this.availableProjects.length > 0
                   ? this.availableProjects.slice(0, 3).join('\n')
                   : t('settings.batchTemplates.editModal.projectsPlaceholder'),
           );
           textArea.inputEl.rows = 3;
           textArea.onChange((value) => {
               this.projects = value
                   .split('\n')
                   .map((line) => line.trim())
                   .filter((line) => line.length > 0);
           });
       });
   ```

3. **`confirm()`**（第 179-194 行）在 candidate 中加入 `projects: this.projects`。

**为什么用 TextArea 而非多选下拉**：项目数量可能较多且动态变化；多行文本输入实现简单、不依赖候选列表完整性（即使项目暂未创建，也可预先填写名称），与原方案一致。`availableProjects` 仅用于 placeholder 提示，不强制约束输入。

### 变更 3：选择弹窗按项目过滤

**文件**：`src/ui/batchTaskModals.ts`

1. **`BatchTemplateSelectModal` 构造函数**（第 33-36 行）新增 `currentProject: string` 参数，并在内部计算过滤后的模板：

   ```typescript
   constructor(
       app: Modal['app'],
       templates: BatchTaskTemplate[],
       currentProject: string,
   ) {
       super(app);
       this.currentProject = currentProject;
       this.filteredTemplates = templates.filter((template) =>

       );

   ```
   - 新增私有字段 `private readonly currentProject: string;` 与 `private readonly filteredTemplates: BatchTaskTemplate[];`

2. **过滤函数**（建议放在 `batch-task-template.ts` 作为纯函数，便于测试）：
   ```typescript
   export function isTemplateAvailableForProject(
       template: BatchTaskTemplate,
       projectName: string,
   ): boolean {
       if (!template.projects || template.projects.length === 0) {
           return true; // 空数组=所有项目可用
       }
       return template.projects.includes(projectName);
   }
   ```
   在 `batchTaskModals.ts` 顶部 import 该函数。

3. **`onOpen`**（第 45-85 行）将所有 `this.templates` 引用改为 `this.filteredTemplates`（共 3 处：第 53 行空判断、第 65 行遍历）。

**为什么把过滤函数放纯函数模块**：便于在 `tests/batch-task-template.test.mjs` 中覆盖（项目已有「只测纯函数」的惯例），避免引入 DOM 测试。

```typescript
private resolveAvailableProjectNames(): string[] {
```

## 假设与决策

1. **字段命名**：采用 `projects: string[]`（复数数组），与原方案一致；不使用可选 `projects?`，避免下游到处判空——`normalizeBatchTemplate` 保证字段始终存在。
2. **输入方式**：编辑弹窗用**多行文本框**（每行一个项目名），而非多选下拉/复选框。理由：实现简单、不依赖候选列表完整性、与原方案一致。`availableProjects` 仅用于 placeholder 提示。
3. **过滤位置**：在**选择弹窗**构造时过滤（而非在视图层预先过滤传入）。理由：弹窗自包含、职责清晰；若未来有「显示全部/仅显示当前项目」开关，改动集中在弹窗内。
4. **过滤函数放置**：`isTemplateAvailableForProject` 放在 `batch-task-template.ts`（纯函数模块），便于单元测试，遵循项目「只测纯函数」惯例。
5. **向后兼容**：旧模板数据无 `projects` 字段 → `normalizeBatchTemplate` 自动补 `[]` → 等价于「所有项目可用」，**无需数据迁移**。
6. **空列表提示**：首版复用 `modal.batchSelect.empty`，不新增「当前项目无可用模板」差异化文案，保持最小改动。如需差异化，后续可追加 `modal.batchSelect.emptyForProject`。
7. **不引入候选项目强校验**：用户在编辑弹窗中输入的项目名不强制必须在 `availableProjects` 列表内（允许预填尚未创建的项目名），仅做 trim 与空行过滤。

### 手动验证（在 Obsidian 中）

1. **向后兼容**：升级后打开设置→批量模板，原有模板仍可见、可编辑、可使用。
2. **新建模板-留空**：新建一个模板，所属项目留空 → 在任意项目右键「批量创建任务」时，该模板均出现在选择列表。
3. **新建模板-指定项目**：新建一个模板，所属项目填写「项目A」→
   - 在「项目A」右键批量创建 → 列表中**有**该模板。
   - 在「项目B」右键批量创建 → 列表中**无**该模板。
4. **多项目**：所属项目填写「项目A」「项目B」（多行）→ 两个项目均可见，项目C不可见。
5. **编辑保存**：编辑已有模板的 projects 后保存，再次打开编辑弹窗，项目文本框内容正确回显；关闭重开 Obsidian 后仍持久化（验证 `areBatchTemplateConfigsEqual` 正确识别变更并落盘）。
6. **非法输入**：projects 输入含空行/前后空格 → 保存后重新打开，空行与空格被清理。
7. **placeholder 提示**：当存在项目文件夹时，编辑弹窗的项目文本框 placeholder 显示真实项目名（前 3 个）。

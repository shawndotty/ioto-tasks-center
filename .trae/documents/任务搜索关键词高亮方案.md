# 任务搜索关键词高亮方案

## 需求

当用户在任务中心搜索任务时（搜索已支持匹配任务笔记的 `content`），点击任务列表中的任务打开笔记后，希望能**高亮显示笔记中对应的关键词**，方便用户快速定位。

## 当前状态分析

- `taskSearchQuery` 存储在 [iotoTasksCenterView.ts](file:///Users/johnny/Documents/Sync/IOTO-Plugins/.obsidian/plugins/ioto-tasks-center/src/views/iotoTasksCenterView.ts#L162)（public 属性），仅用于在任务中心面板内过滤列表

- 该查询**从未传递**给打开的文件

### 任务打开流程
- 点击任务行 → [task-row-renderer.ts](file:///Users/johnny/Documents/Sync/IOTO-Plugins/.obsidian/plugins/ioto-tasks-center/src/views/tasks-center/task-row-renderer.ts#L212) 调用 `view.openTaskFile(task)`
- `openTaskFile` ([iotoTasksCenterView.ts#L1162](file:///Users/johnny/Documents/Sync/IOTO-Plugins/.obsidian/plugins/ioto-tasks-center/src/views/iotoTasksCenterView.ts#L1162)) 委托给 `openFileInPreview(file)` ([#L1287](file:///Users/johnny/Documents/Sync/IOTO-Plugins/.obsidian/plugins/ioto-tasks-center/src/views/iotoTasksCenterView.ts#L1287))
- `openFileInPreview` 调用 `leaf.openFile(file, { active: true })`，**不携带任何搜索/高亮上下文**
- 文件在 preview leaf（垂直分割面板）中打开

### 已有的"定位到文件内位置"先例
- `openTaskFileAtChecklist` ([#L1187](file:///Users/johnny/Documents/Sync/IOTO-Plugins/.obsidian/plugins/ioto-tasks-center/src/views/iotoTasksCenterView.ts#L1187)) 用 `leaf.setViewState` 打开后，通过 `view.editor.setSelection()` 定位光标到特定行/字符，并 `editor.focus()`
- 这是代码库中唯一对文件内位置进行定位的先例

### Obsidian API 关键约束
- Obsidian 公共 `Editor` API **没有内置搜索高亮方法**（无 `editor.search()`）
- `app.internalPlugins.plugins['global-search'].instance.openGlobalSearch(query)` 是**全局搜索**，会离开当前笔记，不适合
- 可行的方案只有：**用 Editor API 手动定位/选中第一个匹配**，或**访问底层 CodeMirror 实例**做高亮

## 方案评估

### 方案 A：定位并选中第一个匹配（推荐）

**原理**：打开文件后，在 `task.content` 中查找关键词第一次出现的位置，用 `editor.setSelection()` 选中它并 `scrollIntoView`。

**优点**：
- 实现简单，约 40 行代码
- 使用公共 Editor API（`getValue` / `setSelection` / `scrollIntoView`），稳定可靠
- 与现有 `openTaskFileAtChecklist` 模式一致
- 无需新增依赖

**缺点**：
- 只能"选中"第一个匹配，不能高亮所有匹配项（除非用户按 Cmd+F 继续）
- 选中 ≠ 高亮（视觉上是蓝色选区，不是黄色高亮）

### 方案 B：触发 Obsidian 内置 in-note 搜索面板

**原理**：打开文件后，模拟用户按 `Cmd+F`（editor:search` 命令），然后将搜索词填入 Obsidian 的 in-file 搜索框。

**优点**：
- 真正的"高亮所有匹配 + 导航"体验（Obsidian 原生搜索框）
- 用户可继续上下翻页

**缺点**：
- Obsidian **没有公共 API** 填入搜索词。`app.commands.executeCommandById('editor:search')` 只能打开搜索框，无法填入关键词
- 需要访问未文档化的内部 API 或直接操作 DOM（`.workspace-leaf-content[data-mode="source"] input.document-search-input`），脆弱且可能随版本失效
- preview leaf 默认是**阅读视图**，in-file 搜索只在编辑视图可用，需要强制切换到 source 模式

### 方案 C：CodeMirror 底层 search addon

**原理**：通过 `editor.cm`（CodeMirror 实例）调用 search addon 的高亮 API。

**优点**：
- 真正的高亮所有匹配

**缺点**：
- `editor.cm` 是未文档化的内部属性
- CM5（桌面旧）和 CM6（移动/新桌面）API 完全不同，需要双分支
- 极度脆弱，Obsidian 版本升级极易破坏

---

## 推荐决策：方案 A

**理由**：
1. **用户原话**："当用户点击时，直接把搜索关键词传入到 Obsidian 自带的笔记搜索功能的搜索框中" —— 但经调研，Obsidian 没有公共 API 支持"填入搜索框"，方案 B 需要操作 DOM 或内部 API，风险高
2. 方案 A 用公共 API 实现"定位到第一个匹配并选中"，是**最稳定**的做法，且改动最小
3. 视觉上，选中第一个匹配 + 自动滚动到该位置，已经能帮用户快速定位

```ts
async openFileInPreview(file: TFile): void {
    this.openingTaskPath = file.path;
    this.render();
    try {
        const leaf = this.ensurePreviewLeaf();
        await leaf.openFile(file, { active: true });
        this.previewLeaf = leaf;
        this.openedTaskPath = file.path;
        if (this.selectedProject) {
            this.lastOpenedTaskByProject.set(this.selectedProject, file.path);
        }

        // 新增：如果有搜索关键词，定位到第一个匹配
        const query = this.taskSearchQuery.trim();
        if (query) {
            await this.scrollPreviewToFirstMatch(leaf, file, query);
        }
    } finally {
        this.openingTaskPath = null;
        this.render();
    }
}
```

**新增辅助方法 `scrollPreviewToFirstMatch`**：

```ts
private async scrollPreviewToFirstMatch(
    leaf: WorkspaceLeaf,
    file: TFile,
    query: string,
): Promise<void> {
    const view = leaf.view;
    if (!(view instanceof MarkdownView) || view.file?.path !== file.path) {
        return;
    }

    const editor = view.editor;
    const content = editor.getValue();
    const normalizedQuery = query.toLocaleLowerCase();
    const normalizedContent = content.toLocaleLowerCase();
    const index = normalizedContent.indexOf(normalizedQuery);
    if (index === -1) {
        return;
    }

    // 将字符偏移转换为 { line, ch } 位置
    const pos = editor.offsetToPos(index);
    const endPos = editor.offsetToPos(index + query.length);
    editor.setSelection(pos, endPos);
    editor.scrollIntoView({ from: pos, to: endPos }, true);
}
```

**注意**：
- 使用公共 API：`editor.getValue()`、`editor.offsetToPos()`、`editor.setSelection()`、`editor.scrollIntoView()`
- 大小写不敏感匹配（与 `matchesTaskSearchQuery` 一致）
- `MarkdownView` 需要从 obsidian 导入（文件顶部已有）

### 文件 2: 无需修改其他文件

- `task-row-renderer.ts` 的点击处理器无需改动（仍调用 `view.openTaskFile(task)`）
- `taskSearchQuery` 已是 public 属性，直接可读

### 边界情况处理
- **空查询**：`taskSearchQuery` 为空时不做任何定位（保持现有行为）
- **阅读视图模式**：preview leaf 可能是 reading 模式，此时 `view instanceof MarkdownView` 为 true 但 `view.editor` 可能不可用。需验证：如果 editor 不可用，静默跳过（`scrollPreviewToFirstMatch` 中的 `instanceof` 检查 + try/catch 兜底）
- **文件已打开**：`shouldSkipOpeningTask` 会跳过重复打开。此时不触发高亮（用户点击的是已打开的文件）。如果用户希望即使文件已打开也重新定位，需要在 `openTaskFile` 的 skip 分支也调用定位逻辑 —— **暂不处理**，保持简单

## 假设与决策

1. **假设**：preview leaf 打开后 `leaf.view` 立即可用且 `editor` 已初始化。`openTaskFileAtChecklist` 已验证此模式可行（它也是 openFile 后立即操作 editor）。
2. **决策**：不为该功能新增设置开关。只要 `taskSearchQuery` 非空就自动定位，行为直观。
3. **决策**：不强制切换到 source 模式。如果当前是 reading 模式，定位可能不生效，但不影响正常打开文件。
4. **决策**：不实现"高亮所有匹配"。如用户需要，可在选中第一个匹配后手动按 Cmd+F。

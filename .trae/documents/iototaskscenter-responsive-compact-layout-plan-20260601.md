# 任务中心窄宽度自适应计划

## Summary

目标是在任务中心所在 tab 宽度较小时，改善项目列表与任务列表的展示方式。

已确认的方向：

- 当任务中心整体宽度小于 `720px` 时，不直接并排展示左右两栏
- 采用“隐藏左侧项目列表 + 在右侧任务列表区提供项目切换入口”的方案
- 这样既能满足小宽度下聚焦任务列表，又不会失去切换项目的能力

本次不包含：

- 改动任务数据结构
- 改动项目或任务的排序/分组逻辑
- 新增设置项让用户自定义断点

## Current State Analysis

### 1. 当前视图结构是固定的左右两栏

文件：

- `src/views/iotoTasksCenterView.ts`

已确认现状：

- `render()` 当前总是创建两个 pane：
  - `ioto-tasks-center__pane--projects`
  - `ioto-tasks-center__pane--tasks`
- 左侧 pane 负责项目列表、项目创建
- 右侧 pane 负责任务列表、搜索、筛选、分组等
- 没有任何“当前宽度模式”的视图 state
- 也没有项目切换菜单或小屏专用入口

结论：

- 如果只在样式层把左栏隐藏，用户会丢失切换项目的入口
- 必须在右侧任务区增加一个项目切换入口，才能让“小于 720px 隐藏左栏”可用

### 2. 当前样式已经存在一个 900px 以下的单列响应式规则

文件：

- `styles.css`

已确认现状：

- `.ioto-tasks-center` 默认是双列 grid：
  - `minmax(220px, 28%)`
  - `minmax(0, 1fr)`
- 在 `@media (max-width: 900px)` 下，当前会改成单列：
  - `grid-template-columns: 1fr`
- 但该规则只是把布局改成一列：
  - 没有隐藏项目列表
  - 没有提供小屏切换项目的替代入口

结论：

- 现有 `900px` 规则会与新的 `720px` 方案直接相关
- 执行时需要统一这两个断点逻辑，避免：
  - 900 以下先变单列
  - 720 以下又切另一套模式
  导致交互和视觉混乱

### 3. 当前代码里没有尺寸监听或响应式状态同步能力

文件：

- `src/views/iotoTasksCenterView.ts`
- `src/` 全局搜索结果

已确认现状：

- 当前仓库中没有：
  - `ResizeObserver`
  - `matchMedia`
  - 宽度变化监听
- 响应式完全依赖 CSS

结论：

- 如果采用“单纯隐藏左栏 + 右侧一直显示项目切换器”的方案，可以主要靠 CSS + 普通 DOM 渲染完成
- 如果要做到“仅小于 720px 才显示项目切换器”，则可通过：
  - 始终渲染入口，交给 CSS 控制显示
  的方式避免引入额外的 JS 尺寸监听

### 4. 当前已有合适的菜单交互模式可复用

文件：

- `src/views/iotoTasksCenterView.ts`

已确认现状：

- 已有 `Menu` 用法：
  - `showTaskCreationMenu()`
  - `showTaskPresentationMenu()`
- 项目切换目前只通过左侧项目列表按钮实现

结论：

- 小屏项目切换器最适合复用 `Menu`
- 这样实现成本低，也与当前插件内交互风格一致

## Proposed Changes

### 1. 在任务列表区域新增“小屏项目切换器”入口

文件：

- `src/views/iotoTasksCenterView.ts`

变更内容：

- 在右侧任务区域标题附近新增一个项目切换入口
- 推荐位置：
  - 任务列表标题下方、当前项目说明文案上方
  - 或与标题同一 header/action 区并列
- 入口触发后弹出 `Menu`，列出当前可见项目

推荐呈现方式：

- 按钮文案类似：
  - `切换项目`
  - 或 `当前项目：xxx`
- 当前选中项目在菜单中可识别

原因：

- 这是在小宽度下隐藏左栏后保留核心导航能力的最小改动方案

实现要点：

- 入口可以始终渲染，但默认仅在小于 `720px` 时显示
- 点击菜单项后复用现有 `selectProject()` 切换项目
- 需要在项目不可切换的状态下禁用，例如：
  - 项目正在加载
  - 没有可见项目

### 2. 引入“小屏紧凑模式”类名，而不是完全依赖多套媒体查询拼接

文件：

- `src/views/iotoTasksCenterView.ts`
- `styles.css`

变更内容：

- 在根容器或 pane 容器层，加入清晰的样式类约定，例如：
  - `.is-compact`
  - 或 `.ioto-tasks-center--compact`
- 该类对应“小于 720px 时仅显示任务列表主区”的视觉模式

推荐实现方式：

- 由于当前没有尺寸监听，优先采用纯 CSS 方式：
  - 直接在 `@media (max-width: 720px)` 中控制
- 若执行阶段发现必须依赖 DOM 结构切换，再考虑轻量 JS state，但不作为首选

原因：

- 当前已有 `900px` 媒体查询，若继续零散追加样式，后续维护成本会上升
- 用一个明确的“紧凑模式”概念可以让样式更容易读懂

### 3. 统一 900px 与 720px 两个断点的职责

文件：

- `styles.css`

变更内容：

- 重新定义响应式策略，避免现有 `900px` 规则与新需求冲突

建议断点职责：

- `> 900px`
  - 维持当前双列布局
- `721px - 900px`
  - 继续使用双列，或改为更紧凑的双列比例
  - 重点是不要过早让左栏占满整行
- `<= 720px`
  - 进入紧凑模式
  - 隐藏左侧项目列表 pane
  - 只显示右侧任务 pane
  - 显示“小屏项目切换器”

原因：

- 用户明确指定的是 `720px` 阈值
- 当前 `900px` 单列策略会让 720 以下仍先呈现“项目列表在上、任务列表在下”的旧行为，不符合新的目标

实现要点：

- 执行时要么删除旧 `900px` 单列规则
- 要么把它改造成仅调整间距/比例而不是改为单列

### 4. 为小屏模式补充样式

文件：

- `styles.css`

变更内容：

- 新增或调整以下样式：
  - 小屏项目切换按钮
  - 小屏模式下隐藏左侧项目 pane
  - 小屏模式下右侧任务 pane 占满宽度
  - 小屏模式下按钮和说明文案的换行/对齐

重点样式点：

- 项目切换按钮应在小屏下明显可见
- 任务区顶部的按钮区不要因为宽度变窄而挤压过度
- 搜索框、tab 栏、设置按钮在小屏下仍需可用

原因：

- 新增入口不能只是“能点”，还要在窄宽度下可读、可操作

### 5. 让小屏模式与当前项目说明、任务搜索、分组折叠共存

文件：

- `src/views/iotoTasksCenterView.ts`
- `styles.css`

变更内容：

- 保证以下现有功能在小屏模式下仍正常：
  - 当前项目说明文案
  - 任务搜索
  - 筛选 tab
  - 任务列表设置按钮
  - 分组折叠
  - 优先级 badge

原因：

- 小屏模式的目标是“聚焦任务列表”，而不是阉割现有任务操作能力

实现要点：

- 小屏切换项目后，任务列表应立即刷新
- 切换项目不应影响当前搜索词、排序分组设置等既有状态规则

### 6. 执行阶段进行手动验证

文件：

- 无新增源码文件，仅为执行阶段验证范围

验证重点：

- 当 tab 宽度大于 `720px` 时，仍是正常双列布局
- 当 tab 宽度小于等于 `720px` 时：
  - 左侧项目列表不显示
  - 右侧任务列表占满宽度
  - 可通过右侧项目切换器切换项目
- 切换项目后任务列表正确刷新
- 小屏下搜索、tab、设置按钮、分组折叠仍可正常使用
- 宽度在 `720px` 临界点附近切换时，布局不会明显错乱

## Assumptions & Decisions

- 小于 `720px` 时采用“隐藏左栏 + 项目切换器”方案，而不是完全隐藏左栏后不提供入口
- 项目切换器使用 `Menu` 实现，复用现有交互模式
- 断点以用户要求的 `720px` 为准
- 现有 `900px` 响应式规则需要收敛，不能继续保留与新方案冲突的单列行为
- 小屏模式只改变布局与项目切换入口，不改变任务列表现有业务逻辑

## Verification Steps

1. 大屏验证
   - 大于 `720px` 时保持原有双列布局

2. 小屏布局验证
   - 小于等于 `720px` 时隐藏左侧项目列表
   - 右侧任务列表独占可用宽度

3. 小屏交互验证
   - 右侧能看到项目切换入口
   - 可切换到任意可见项目
   - 切换后任务数据正确刷新

4. 功能兼容验证
   - 搜索功能正常
   - tab 筛选正常
   - 设置按钮正常
   - 分组折叠正常
   - 优先级显示正常

5. 响应式边界验证
   - 在 720px 左右来回调整宽度时，布局切换平稳
   - 不会出现项目列表与任务列表同时错位或重叠

6. 自动化与静态检查
   - 运行 `npm test`
   - 运行 `npm run build`
   - 运行 `npm run lint`
   - 对修改文件执行 diagnostics 检查
